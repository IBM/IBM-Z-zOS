    New in this Release

    Version 1.6.3:

      The new ZAWARE option produces a report of the messages that meet zAware training criteria.

    Version 1.6.2:

      The new REPORT CNCM option produces a report, by console name, of the number of commands issued by the console and the number of messages queued to the console by console name. For consoles used by automation, this can be an indication of the level of automation activity.

    Version 1.6.1:

    • This release of the Message Analysis Program introduces significant new function: IPL start-up statistics. The new IPLSTATS input option enables this new support and causes the new IPL start-up report to be produced if the new IPLST DD and IPLSQ DD statements are also provided. The messages listed in the new IPL start-up report can consist of all start-up messages or just the specific messages used to generate the start-up statistics using the NOFILTER or FILTER options. A separate report with all of the message event times normalized to the beginning of the IPL is produced through the IPLSQ DD if the IPLSTATS LIST option is specified. A new IPLID input option allows an optional 8 character identifier to be included on each line of the IPL sequence report. This is helpful if the IPL sequence report is used as input to the IPLMERG program which can be used to compare the events occurring in two IPL SYSLOGs.

    • The new USESYS input option allows the Message Analysis Program to filter the input SYSLOG for records from a particular system name. This is useful when the SYSLOG has been created from an OPERLOG and contains records from multiple systems.

    • The new USESYS input option limits the records that are written to the OUT DD to records from the specified system name. This allows system-specific SYSLOG subsets to be created from a SYSLOG that was created from an OPERLOG containing records from multiple systems. As part of this support, the system name is guaranteed to be on all records, including all lines of multi-line messages.

    • The new USESTC input option allows the Message Analysis Program to filter the input SYSLOG for records from a particular started task name of the form STCnnnnn.

    • The new USESTC input option limits the records that are written to the OUT DD to records from the specified started task name. This allows started task specific SYSLOG subsets to be created from a SYSLOG that was created from an OPERLOG containing records from multiple systems. As part of this support, the started task name is guaranteed to be on all records, including all lines of multi-line messages.

    • The SAMPLSTR and SAMPLEND input options now limit the records that are written to the OUT DD. This allows a SYSLOG subset to be produced for a specific time period.

    • The record offset and record adjustment input options now affect the records that are written to the OUT DD. This allows the SYSLOG to be put into a standardized form so that no adjustments are required on any subsequent processing.

    • Records written to the OUT DD will now be guaranteed to contain date and timestamp information on every record (including all lines of a multiple line message), allowing the file to be sorted. This can be helpful if the SYSLOG was produced from an OPERLOG containing messages from multiple systems since the messages in the merged log are not always in strict time sequence. This also allows multiple SYSLOGs to be merged together in time sequence.

    • The IGNORE option now supports partial message IDs. For example, if you want to ignore all "SEL" prefix messages (messages of the form SELnnnnn where "n" is a numeric digit), you can now specify IGNORE(SEL) and all SEL prefix messages will be ignored.

    • The new YEAR4 option performs all of the adjustments necessary to compensate for use of 4-digit year fields in the SYSLOG. YEAR4 is a "shorthand" for specifying LOGDATE(21), LOGTIME(27), LOGJOB(39), LOGREQFL(48) and LOGMSG(57). YEAR4 should be used instead of specifying these options individually, especially if you are producing an IPL statistics report (the IPLSTATS option), since 4-digit year fields affect the processing of certain messages. YEAR4 makes the processing of these messages aware that 4-digit year fields were in use; use of the individual field adjustment options does not.

    • The "Messages by Component-ID" report can now handle component IDs up to 7 characters in length.

    Introduction

    "95% of anything is junk. (1) " -- Sturgeon's (2) Law.

    Sturgeon's Law applies to messages. The Message Analysis Program will help you identify:

    • what messages (and commands) are being generated in your environment

    • who's generating them (and at what rate)

    • where are they going

    • what messages are being suppressed

    • what messages are being processed (and in some cases, what is being done to them)

    Program Options

    The Message Analysis Program allows a great deal of customization. However, it is strongly recommended that you use the program's defaults until you are more familiar with the program's facilities.

    If there is an "OPTIN" DD statement, the file will be opened and program control parameters will be read from there.

    Program options must be entered from a data set or as part of an in-stream JCL file.

    • OPTIN records must be 80 characters in length although only the first 71 characters are used as input

    • 8-character line numbers may be used (columns 72-80)

    • Each option must begin in column 1 of a record

    • Options may not be continued on the next record

    • Parameters contained within parentheses must not contain blanks (with the exception of IGNORE).

    • Multiple parameters contained within parentheses must be separated by commas.

    If no options are provided, then the program default functions will be performed based on the input and output data sets which are provided.

    All options specified are listed in the "OPTIONS IN EFFECT" report at the beginning of the output. If an input option could not be recognized, the option will be printed in the OPTIONS IN EFFECT list and will be followed on the next line by an appropriate error message. The requested operation will not be performed and defaults, if any, will be used.

    The error messages produced are as follow:

    • UNKNOWN PARAMETER

      Its not a parameter that the program knows anything about. Typically a legitimate parameter that has been mis-spelled.

    • TABLE FULL

      The program is expecting no more than some number of items and you've exceeded that number. Check the documentation and reduce the number of items specified.

    • SYNTAX ERROR

      You have not specified the parameter properly. Typically a problem with illegal blanks, missing parentheses, missing commas, etc.

    • VALUE OUT OF RANGE

      A numeric value is either smaller or larger than the program allows. Check the documentation and specify a larger or smaller value as appropriate.

    • UNKNOWN OPTION

      Its not an option that the program knows anything about. Typically a legitimate option that has been mis-spelled or does not begin in column 1.

    • INVALID VALUE

      The value specified is a not a valid choice. For example, the program expected to find a non-blank character but found a blank.

    • LENGTH ERROR

      The value specified is too long. For example, the program expected to find a string 1 to 8 characters in length but found one 9 characters in length.

    • DUPLICATE TEMPLATE template

      The message ID template duplicates a previously processed template or is a proper subset of a previously processed template and cannot be processed.

    • TEMPLATE NOT PROCESSED template

      The message ID template was not processed because the limits on the number of templates that can be specified was exceeded.

    • EXCESSIVE MESSAGE TEMPLATES SPECIFIED

      The program is terminated because the limits on the number of templates that can be specified was exceeded. This is usually due to an excessive number of specific message templates. This error is a more serious manifestation of the previous error.

    Comment Statement

    Comment statements may be placed anywhere in the option input stream after the TITLE statement. A comment is signified by placing an asterisk (*) in column 1. Comments are printed as part of the "OPTIONS IN EFFECT" report at the beginning of the output.

    All of the option statements (with the exception of TITLE) may have comments placed on them. The comment must be separated from the statement specification by at least one blank.

    TITLE Statement

    Titling information may be supplied by coding:

    TITLE:
    followed by up to 65 characters of titling information
    If it is used, TITLE: must be the first option specified. Titling information is printed at the beginning of each report that is produced.

    LOGTYPE Statement

    The user may specify whether the log being read is a 'SYSLOG' or a SYSLOG containing a JES3 'DLOG'. The default is 'SYSLOG'.

    LOGTYPE(xxxxxx)
    where "xxxxxx" is either:
    • SYSLOG - for an MVS/JES2 SYSLOG -or-
    • DLOG - for an MVS/JES3 DLOG

    In general, it is better to not specify LOGTYPE and allow the program to determine the log type itself. The program can handle mixed format logs containing both SYSLOG format records and JES3 DLOG format messages.

    Note: You may wish to use the 'PREVIEW' DD (see "PREVIEW DD (Optional)") to establish how well fields in the log (especially if it is a JES3 DLOG) align with their expected positions. Use of the field-aligning operations (OFFSET, DLGDEST, etc.) discussed in "Field-Alignment Statements" may be required. :enote.

    USESYS Statement

    The USESYS input option allows the Message Analysis Program to filter the input SYSLOG for records from a particular system name. This is useful when the SYSLOG has been created from an OPERLOG and contains records from multiple systems.

    USESYS(sysname)
    where "sysname" is the 1 to 8 character system name to be searched for.

    As part of this support, the system name is guaranteed to be on all records, including all of the lines of multi-line messages.

    The USESYS input option limits the records that are written to the OUT DD to records from the specified system name. This allows system specific SYSLOG subsets to be created from a SYSLOG that was created from an OPERLOG containing records from multiple systems.

    USESTC Statement

    The USESTC input option allows the Message Analysis Program to filter the input SYSLOG for records from a particular started task name.

    USESTC(stcname)
    where "stcname" is a started-task name of the form STCnnnnn and "nnnnn" is a 5 digit numeric value.

    As part of this support, the started task name is guaranteed to be on all records, including all of the lines of multi-line messages.

    The USESTC input option limits the records that are written to the OUT DD to records from the specified started task name. This allows started task specific SYSLOG subsets to be created from a SYSLOG.

    CMDPREFIX Statement

    The user may specify the command prefix characters that the program is to use to identify JES2, JES3 and DB2 commands.

    CMDPREFIX subsys(char),subsys(char),subsys(char)
    where "subsys(char)" is:

    • JES2(x) - where "x" is the JES2 command prefix character. The default is the $ (dollar sign) character.

    • JES3(xy) - where "x" is the JES3 command prefix character and "y" is the command prefix synonym character. The default is the * (asterisk) with the 8 (eight) character as its synonym.

    • DB2(x) - where "x" is the DB2 command prefix character. The default is the - (minus) character.

    It is not necessary to specify this statement if the default command prefix characters are being used.

    Any or all of the subsystems can be specified, in any order.

    The first subsystem specification must be separated from the word CMDPREFIX by a single blank; subsequent subsystem specifications must be separated by a single comma.

    The blank character is not a legitimate command prefix character. If specified, an INVALID VALUE error will result.

    CPFPREFIX Statement

    The user may specify the command prefix strings that the program will use to identify instances of DB2 and other subsystems which use the operating system's Command Prefix Facility (CPF).

    CPFPREFIX /string/.../string/
    where:

    • / is any non-blank delimiting character. The delimiting character must be separated from the word CPFPREFIX by one blank and must be the first and last character in the specification. You must use the delimiting character consistently within this CPFPREFIX statement but you may use another delimiting character on a subsequent CPFPREFIX statement.

    • string is any valid command prefix string, from 1 to 8 non-blank characters in length. You may specify as many command prefix strings on a single statement as will fit. You may specify as many CPFPREFIX statements as needed. A total of 64 strings may be specified across all of the statements. If more than 64 strings are specified, an error message is produced and only the first 64 command prefix strings are used.

    The following statements were used to identify the 12 instances of DB2 in a data-sharing group in a Parallel Sysplex environment: 

    CPFPREFIX /@DBA1/@DBB1/@DBC1/@DBD1/@DBE1/
CPFPREFIX /@DBF1/@DBG1/@DBH1/@DBI1/@DBZ1/
CPFPREFIX /@DB81/@DB91/

    Command prefixes are also used as a "shorthand" way of routing commands to specific systems in a sysplex environment (see discussion of the IEECMDPF sample program in SYS1.SAMPLIB). For example, if the sysplex consists of systems SY1 through SY8 and these names have been defined to the operating system through the IEECMDPF program, then all of the following are equivalent ways of routing a command: 

    ROUTE SY1,command
SY1 command
SY1command

    For the Message Analysis Program to recognize the "command" part of the command, the following CPFPREFIX statement would be required: 

    CPFPREFIX /SY1/SY2/SY3/SY4/SY5/SY6/SY7/SY8/

    The Computer Associates Multi-Image Manager (MIM) provides a similar command shorthand using a subsystem identification character as a prefix: 

    .a command
    where the period is the subsystem identification character and "a" is a system name.

    For the Message Analysis Program to recognize the "command" part of the command, the following CPFPREFIX statement was used: 

    CPFPREFIX /.ALL/.A/.B/.C/.D/.E/.F/.G/.K/.Y/

    Since .A is a proper subset of .ALL, .ALL should be placed in the specification first to insure that .ALL is checked before .A is checked.

    IGNORE Statement

    Messages may be completely ignored by all logic in the program, if so requested. You can specify a specific message ID (such as IEF196I) that you want to ignore or you can specify a message ID prefix (such as SEL) to ignore an entire set (all SELnnnnn messages) of messages. A maximum of -8- message IDs to be ignored (8 specifications of the IGNORE statement) is supported.

    IGNORE(xxxxxxxx)
    where "xxxxxxxx" is a 1-10 character message ID or message ID prefix. The message ID or message ID prefix may be padded by blanks on the right to a total length of no more than 10 characters.

    You may specify a complete message ID (such as IEF196I to remove all instances of IEF196I messages) or a message ID prefix (such as SEL to remove all messages that begin with the characters SEL).

    This statement is useful in removing bursts of a particular message ID that would otherwise skew the results. An example would be bursts of I/O error messages. IGNORE can be used to remove spurious messages.

    IGNORE may also be used to remove common log-only messages (such as IEF196I) from all computations.

    OFFSET Statement

    Each log record read may be offset to the right or left so that text fields may be properly aligned. This program assumes that the 1st character of each log record contains printer control (carriage control) information. If the record does not contain printer control information, then the entire record should be offset 1 character to the right (positive). Offsetting is performed before the record adjustment exit is invoked and before any individual field-alignment operations are performed.

    OFFSET(n)
    where "n" may be a positive (rightward) or negative (leftward) offset. The following numeric forms are supported: n, +n, -n
    The OFFSET statement may also be used to strip off extraneous fields appended to the beginning of each log record by many log archiving utilities. For example, if the record is prefixed by a four-character system identifier as follows (and no printer control), 
    CS01N 0020000 CS01     01003 00:00:01.41 STC27536 00000290  IEF695I START
    the identifier can be stripped by specifying: 
    OFFSET(-3)
    This will treat the last character of the prefix as the printer control character, but that is OK.

    The effects of OFFSET processing can be determined using the PREVIEW capability described in "PREVIEW DD (Optional)".

    Field-Alignment Statements

    Each log record read may be reconstituted before processing to place individual fields into the positions the program expects to find them in. Field-alignment operations are performed after OFFSET processing is performed and after the record adjustment exit is invoked. Any or all of the following statements may be specified. Fields which are not explicitly specified will use default offsets.

    All offsets (column positions) are 0-relative to the first character after the printer control byte.

    The effects of field-alignment processing can be determined using the PREVIEW capability described in "PREVIEW DD (Optional)".

    RATEMSGS Statement

    The messages which are considered during computation of message rates can be explicitly specified in terms of their classifications:

    RATEMSGS(xxx,...,xxx)
    where:
    ALL
    ALL messages are used in the rate computations. This is the default.
    WTP
    Write-To-Programmer messages are used in the rate computations.
    MLG
    Write-To-Log or MLOG-only(JES3) messages are used in the rate computations.
    RRT
    JES3 re-routed messages are used in the rate computation.
    SOL
    Solicited messages are used in the rate computation.
    U/S
    Unsolicited/suppressable messages are used in the rate computation.
    U/U
    Unsolicited/unsuppressable messages are used in the rate computation.
    UNS
    Unseen messages are used in the rate computations.
    KWN
    Known messages are used in the rate computations.
    UNK
    Unknown messages are used in the rate computations.

    Notes:

    • Not all messages have been pre-classified.

    • 'Known' messages encompass all messages that have been classified.

    • 'Unknown' messages encompass all unclassified messages.

    ZAWARE Statement

    Inclusion of the ZAWARE statement in the input parameters will cause production of a report of the message IDs which satisfy the following zAware training message criteria:

    • each unique message ID must have at least 3 instances
    • at least 3 of the instances must come from different 10 minute time intervals

    zAware states that it needs a minimum of 375 unique message IDs for training purposes.

    ADJEXIT Statement

    The user can specify the name of a user-written exit that is to receive control for each record that is processed from the log. The exit can be used to perform adjustments that are beyond the capabilities of the Message Analysis Program's internal record adjustment facilities. The exit sees the entire input record and is able to examine and modify all of the fields contained within it.

    For instance, a user exit could be used to put non-standard messages into standard form. The log analysis program assumes that the first text field in the message portion of the log contains a message ID. There are a number of vendor messages that put either a step name or a task name out as the first field in the message, and follow that with a message ID. A fairly simple user exit could be written to look for such messages and transpose the message ID and stepname / taskname fields. (The message ID should be left-justified in the message and separated from any subsequent text by at least one blank).

    There are also vendor messages that use delimiters other than blanks at the end of message IDs. Again, an exit can be written to scan for these and replace them with blanks.

    The exit can be used to change non-standard message IDs (see "Unknown Message report" for a description of the message ID formats acceptable to the Message Analysis Program) into a standard form so that the program can properly process the message and generate statistics.

    MSGIDS Table

    The MSGIDS table allows you to describe what message IDs in your environment look like so that the program can properly recognize them.

    The following default alphabetic and numeric substitution characters are used: 

       @ - represents any alphabetic character (A-Z)
   # - represents any numeric character (0-9)

    All other characters represent themselves.

    The alphabetic and numeric substitution characters can be redefined at any time by defining new substitution characters, using the ALPHABETIC(@) and NUMERIC(#) specifications, where '@' and '#' can be any special (non-alphabetic, non-numeric) character. The ALPHABETIC(@) and NUMERIC(#) specifications -MUST- begin in column 1. DO NOT have the same special character in use by both the ALPHABETIC and NUMERIC specifications at the same time!

    The following rules must be observed:

    • A message ID template definition MUST begin in column 1.

    • An asterisk (*) in column 1 represents a comment statement.

    • A message ID template MUST NOT contain any imbedded blanks. A blank is considered a delimiter.

    • A comment may follow a template definition, but must be separated from it by at least one blank character.

    • THE MAXIMUM LENGTH OF A MESSAGE ID TEMPLATE IS <10> CHARACTERS.

    • WARNING: AN 0C4 ABEND WILL RESULT IF AN ENTRY IS A PROPER (COMPLETE) SUBSET OF ANOTHER. For example, you cannot specify both @@@#### and @@@####I because the first is a proper subset of the second.

    • WARNING: AN 0C4 ABEND WILL RESULT IF THIS TABLE HAS BEEN IMPROPERLY UPLOADED FROM A PC. The program depends on the '#' character having a hexadecimal value of x'7B' and the '@' character having a hexadecimal value of x'7C'. If these characters have been translated to anything else during an upload operation, an 0C4 error will occur during program initialization. In some countries, substituting an "a" with an accent for the @ character and the British pound character for the # character will fix this problem. 

    +---NOTE---------------------------------------------------------------------+

    There is a limit to the number of message ID template entries that can be specified. This limit depends on the amount of "likeness" in the template definitions when definitions are viewed from left to right. For example, the two definitions: 
    @@@###A
@@@###B
    are less complicated to represent than the two definitions: 
    @@@A###
@@@B###

    In general, it is a good idea to use generic characters as much as possible and to try and keep non-generic characters as far to the right in the definition as possible. It is recommended that templates with suffixes (such as A, E, I, D and W) be made generic entries and if a specific product produces a message that is a proper subset of the generic template that a specific message template be produced for it. A specific template -can- be a proper subset of a generic template but you cannot have a generic template be a proper subset of another generic entry or a specific template be a proper subset of another specific template.

    SYSLOG Record Summary report

    The SYSLOG Record Summary report is derived from the first two characters of a standard format SYSLOG. The first character determines the type of message (this is shown vertically in the report) and the second character shows whether the message is unsolicited or in response to a command (this is shown horizontally in the report). The report is given first in terms of the number of records found in each category and secondly in terms of the percentage (%) of records found in each category.

    Note: The command, command response and internal command columns are only as accurate as the products setting the flags. You may wish to cross-check this information with that in the Commands in Alphabetic Order Report and the Message IDs by Frequency Report.

    The thing to look for here is whether there is a reasonable spread down the first column (titled -BLANK-) or whether all or almost all of the messages are in the "UNKNOWN RECORDS" category. For JES3 DLOGs, this is normal. However, if this occurs with an MVS/JES2 SYSLOG, it indicates one of several possible conditions:

    • The log is a SYSLOG, but fields in it do not line-up with where the program is expecting to find them.

      The most common problem here is that the log does not contain printer control (carriage control) information in the first byte of each record. This can be compensated for by using the OFFSET input parameter to offset each input record by one character to the right: OFFSET(1).

      Another possibility is that the customer has altered the format of the log and has deleted or moved the two record type characters. The input options do not provide a way of compensating for the deletion or movement of these two characters.

    • The SYSLOG is from a pre-SP2.1.2 level of MVS. This version of the Message Analysis program is not designed to accommodate pre-SP2.1.2 SYSLOG formats.

    You should also total up the percentage of messages in the "COMMAND RESPONSE" column. If the total exceeds 20%, you should probably analyze why your operators are issuing so many commands. Are they running the system, or, are they managing it? If there is a lot of command response traffic, it is probably the former.

    LINES FROM UNKNOWN SOURCE are those records that are written directly to the log using the WTL (Write to Log) macro instead of being written to the log via WTO/WTOR. Messages written to the log through WTL are not prefixed with the same fields of information that WTO/WTORs are. The usual culprits are Program Offerings, the Network products (NMPF), and vendor products. The preferred method of putting messages into the log if you don't want them displayed to an operator is through WTO with the "hardcopy only" indicator set.

    RAS-Related Message Summary report

    The RAS-Related Message Summary report is designed to provide a "quick look" capability to assess whether the data in the log provides a reasonable sample.

    The first part of the "quick look" is the number of occurrences of specific messages, typically the number of jobs started/ended, number of operator replys, number of tape or pack mounts, number of job abends, the number of jobs with JCL errors, and so on.

    The second part of the "quick look" divides the number of jobs run into job, started task and time sharing user (TSO) categories.

    The third part of the "quick look" lists the number of different types of records processed. Of interest here is:

    • SLG TOT / SYSLOG RECORDS PROCESSED - the total number of records successfully read by the program.

    • LR SHRT / TOTAL SHORT RECORDS category indicates the number of very short (8 bytes or less) records that were read and ignored. These may be an artifact of a log archival program.

    • UNKNOWN / TOTAL UNIDENTIFIED RECORDS category indicates the number of records that the program could not make any sense out of (see also the previous section's discussion of LINES FROM UNKNOWN SOURCE). If this count is more than 15% or 20% of the SYSLOG RECORDS PROCESSED, there is a lot of message traffic that will not be factored into any of the subsequent reports. Unidentified records are factored into the message rate computations and graphs.

If any particular ABEND code (with the exception of the x22 operator cancel ABENDS) shows a large count, it may indicate that the system was "sick" in some form and that the information in the log is not representative of the system when it is operating normally.

    • The 10 minute histogram will typically have some noise at its left end and then rise to the right. The steepness of the curve will depend on how uniform the operation of the installation is.

    • The 1 hour histogram will also have noise at its left end, and again the curve will depend on how uniformly the installation is operated. Traditional 3-shift operations will have a decided curve; systems serving internationally may have little curve at all since the demand on the computing system does not change much over the period of a 24 hour day.

    • The first column provides the time-of-day that the start-up message or command occurred in hh:mm.:ss.th format. This is helpful if you want to find the message or command that was used to generate the statistics.

    • blank records

    Message Rate graph format

    The horizontal axis shows the instantaneous rate with which messages were created (as opposed to the rate at which they were displayed). The horizontal axis uses a logrithmic scale; each major (|) sub-division represents a decimal order of magnitude (power of 10). The scale ranges from 0.001 (1/1000th) of a message per second (on the left) to 10K (10,000) messages per second on the right. Minor sub-divisions (:) are provided at 2 times and 5 times each power of 10.

    Five different interval durations are plotted on each graph:

    • 1 second (1)
    • 10 second (2)
    • 1 minute (3)
    • 10 minute (4)
    • 1 hour (5)

    For the purposes of discussion, assume that the 1 second interval line intersects the 80% horizontal line at the 10 message per second vertical line.

    For the % of All Messages graph (the first one of the pair), this means that 80% of the messages found on the log occur in busy 1 second intervals having an instantaneous message rate of 10 messages per second or less and 20% of the messages found on the log occur in busy 1 second intervals having an instantaneous message rate of 10 messages per second or more.

    For the % of All Intervals graph (the second one of the pair), this means that 80% of the busy 1 second intervals found on the log have an instantaneous message rate of 10 messages per second or less and 20% of the busy 1 second intervals found on the log have an instantaneous message rate of 10 messages per second or more.

    Points on all of the other lines can be interpreted in similar fashion.

    Message Rate graph usage

    The 1 second interval line will show burst conditions, typically command responses in response to operator queries. If the operators are not in the habit of properly qualifying their queries to minimize the size of the response, significant burst activity can result. We have observed instantaneous message rates in excess of 1000 messages per second for unqualified JES2 $DN ("show me everything in the output queue") command responses.

    At the other extreme, the 1 hour interval line will show long-term, sustained message rates.

    The degree to which the 1 second interval line and the 1 hour interval line spread is a crude measure of the degree to which the installation lets the machine run itself.

    • If the lines fall fairly close together, the machine is pretty much running itself.

    • If the 1 second line diverges substantially from the 1 hour line, this is usually an indication of excessive manual operation of the system. (Depending on how pro-active your automated operations package is, you may observe the same effect).

    The degree to which second and third shifts are a mirror image of first shift will determine the slopes of the sustained message traffic lines (1 hour, 10 minute, 1 minute). If the shifts are pretty much homogeneous (the same type of workload 24 hours a day, 7 days a week), the sustained message traffic lines will be almost vertical except at the extremes.

    All interval curves are affected by the size (duration) and quality of the input sample.

    The fourth pair of graphs is perhaps the most important: it shows the absolute minimum message traffic that will be directed to the Master Console. If the 1 hour interval line (5) on this graph exceeds 1 message per second, then you should seriously consider doing more message suppression.

    Instantaneous Message Rate graph

    The Instantaneous Message Rate graph shows the instantaneous message rate experienced in every busy interval in the log. (A busy interval is one that has at least one message occur in it). The size of the interval reported on can be specified using the IMSGINTV statement. The choices are: 1 second, 10 seconds, 1 minute, 10 minute and 1 hour. The default interval size is 1 second.

    The message rate is presented as a bar graph across the page, with linear, logrithmic or relative scaling (chosen using the IMSGSCAL statement). An appropriate default scale is generated depending on the type of scaling chosen and the interval size chosen. The maximum value that will be scaled on the graph when linear scaling has been requested can be specified by the user using the IMSGRANGE statement. The default scaling is logrithmic. With a log scale, each major division represents another power of 10 (1, 10, 100, 1000, etc). The relative scale runs from 0 to 100 and can be thought of as a percent of the total message rate in the interval. Regardless of the type of scaling chosen, the scale line is repeated every 50 intervals.

    The number of lines in the graph between page-skips can be specified with the IMSGLNECT statement.

    Instantaneous Message Rate graph interval size considerations

    A very short interval size (1 second, 10 second) will allow you to precisely determine when bursts of messages occurred; however, the resulting report can be of considerable length.

    A long interval size (10 minute, 1 hour) will allow you to to observe long-term sustained message rates and observe how this long-term sustained message rate varies with time of day and date. Long interval sizes will tend to smooth and possibly obscure message burst conditions. On the other hand, the reports are typically fairly short.

    Instantaneous Message Rate graph scaling considerations

    Linear scaling will show you the instantaneous message rate exactly as it occurred, without distorting the absolute magnitudes of burst periods or periods of quiet. Linear scaling also makes it easier to compare the absolute magnitudes of message traffic being suppressed or routed to the Master Console. A disadvantage is that the maximum value chosen for linear scaling purposes will probably be chosen to accommodate burst conditions. Since the message rates during burst conditions are often 2 to 3 decimal orders of magnitude (100x to 1000x) greater than the steady-state message rate, the steady-state message traffic is often reduced to a flat line at the bottom of the graph. (This is much less of a problem as the interval size is increased). If the maximum value chosen for scaling is chosen to be less than the maximum message burst rate observed, truncation of the maximum burst rate displayed will occur. However, the steady-state message rate may then encompass more of the graph.

    Logrithmic scaling will guarantee that both steady-state and burst condition message rates can be shown on the same graph without reducing one to insignificance or resulting in the truncation of the other. However, logrithmic scales are more difficult to interpret and it is far more difficult to compare the absolute magnitudes of quiet passages and burst conditions. Quiet passages appear more pronounced than they really are and burst conditions do not appear as significant as they really are.

    Relative scaling is designed to show proportions rather than absolute values. It will show, of the total message traffic, that proportion that is suppressed, unsuppressed or routed to the Master Console. It does not show (graphically) whether the message traffic is being produced during a period of intense activity or during a period of very light activity.

    • of the 20, 5 messages were bound for the Master Console (represented by the "="s and "M"s)

    • of the 5 bound for the Master Console, 3 messages (represented by the "="s) were not suppressed

    Instantaneous Message Rate graph interpretation

    The following configurations commonly occur and have the following interpretations: 

         88067 07:39:20      20 =====----AAAAAAAAAAA

    In this case, no "M" characters are visible (they have been completely overlayed by the "=" characters), indicating that the number of messages present on the Master Console after message suppression is taken into account is the same as the number of messages present on the Master Console prior to message suppression. This is typically not desirable unless the number of messages routed to the Master Console is already low (this would be indicated by a very short bar of "=" characters). The longer the bar of "A" characters, the better. 

         88067 07:39:20      20 =====----MMAAAAAAAAA

    In this case, "M" characters are obscuring either "-" or "A" characters, so it is not possible to precisely determine the effectiveness of message suppression against all message traffic. If the "M" characters are obscuring only "-" characters, then message suppression has been less effective against all messages; if the "M" characters are obscuring only "A" characters, then message suppression has been more effective against all messages. The longer the bar of "A" characters, the better. From this configuration, it is possible to accurately determine the effectiveness of message suppression on traffic destined for the Master Console. 

         88067 07:39:20      20 =====MMMMMMAAAAAAAAA

    In this case, no "-" characters are visible (they have been completely overlayed by the "M" characters). The "M" characters may also be overlaying some of the "A" characters, so it is not possible to tell how effective message suppression has been against all of the message traffic. From this configuration, it is only possible to ascertain the effectiveness of message suppression on message traffic to the Master Console. The longer the bar of "A" characters, the better.

    Instantaneous Message Rate graph usage

    Linear and logrithmic scaling are useful when you want to identify when burst conditions or periods of inactivity occurred. It is easy to spot such periods visually, and using the date and time stamps, find the corresponding message traffic in the log.

    Used with larger interval sizes, linear and logrithmic scaling can be used to compare message rate activity as a function of time of day and date.

    Relative scaling gives you a good measure of how message suppression and message routing strategies vary by time. Ideally, the proportion of messages suppressed and/or routed to the Master Console should be relatively invariant. In practice, the amount of messages suppressed and routed to the Master Console will often vary by time of day and by message rate. Indeed, many times the fraction of messages suppressed and the number of messages not routed to the Master Console will decline during periods of high message rates, which is exactly the opposite of what is desired. Relative scaling then, can show you when your message suppression, message automation and message routing strategies are less than completely robust. The date and time stamp fields will allow you to find the corresponding periods in the log.

    Time Used Report

    The Time Used Report reports on the amount of processing time used within various parts of the Message Analysis Program. The report can be used to diagnose performance problems, some of which can be dealt with by adjusting your usage of the Message Analysis Program.

    The following example is taken from the processing of a 600K record SYSLOG: 

    MSGLG610               0 hours  0 minutes  6.823455 seconds
 | ADJDATA             0 hours  0 minutes  2.583939 seconds
 | | USEREXIT          0 hours  0 minutes  0.000000 seconds
 | PARSE               0 hours  0 minutes  2.147488 seconds
 | | UKMSGS            0 hours  0 minutes  0.599752 seconds

    Note that the times are nested: the MSGLG610 time includes all of its own time as well as all of the time attributable to ADJDATA and PARSE. The ADJDATA time includes all of its own time as well as all of the time attributable to the USEREXIT. The PARSE time includes all of its own time as well as all of the time attributable to the UKMSGS.

    Typically, the time that is unique to MSGLG610 (the MSGLG610 time minus the ADJDATA time and PARSE time) is fairly small. MSGLG610 time includes:

    • reading in the options
    • initializing data structures
    • reading in tables
    • reading the SYSLOG records and performing any filtering on those records
    • producing all of the reports

    The time that is unique to ADJDATA (the ADJDATA time minus the USEREXIT time) is usually fairly small. ADJDATA time includes:

    • SYSLOG record offset processing
    • SYSLOG record field alignment processing
    • preview processing

    Obviously the more offset and/or field alignment processing that you do, the larger the amount of time that will be spent in ADJDATA processing. Some amount of time will always be spent in this routine even if no explicit processing is requested.

    USEREXIT time is the time spent within the customer-written message processing exit. Depending on what you are doing and how efficiently you are doing it this time can be small or large. In the example, no user exit was used and no time was spent in it.

    PARSE is the main processing routine of the Message Anaylsis Program. The time that is unique to PARSE (the PARSE time minus the UKMSGS time) is usually proportionate to the MSGLG610 time. Both tend to be proportionate to the number of SYSLOG records processed. PARSE time includes:

    • identifying valid message IDs
    • dealing with non-contiguous multi-line messages
    • building and updating the message data structure
    • collapsing command variations
    • building and updating the command data structure
    • building and updating message rate data structures

    UKMSGS processing handles all of the messages which do not have identifiable message IDs. The time spent in UKMSGS can increase or decrease substantially depending on how accurately message IDs are decribed in the MSGIDS table (see "MSGIDS Table"). In the example shown, the default MSGIDS table shipped with the Message Analysis Program was used. Identifying a message with a MSGIDS table rule is substantially faster than handling the message through unidentifiable message ID processing.

    Errors

    The Message Analysis Program has been in use in various versions for more than thirty years so many error situations have already been encountered and fixed. That is not to say that the program is absolutely bug free!

    What errors do still occur are usually associated with faulty input data although the symptoms may not seem related. In particular, 0C4 and 0C9 errors can be caused by input which does not have fields located in the positions that the program expects.

    0C1 Errors

    0C1 errors typically result from a failure to specify all of the necessary output DD statements. This can occur when specifying REPORT(ALL) since some reports are written to specific optional DD statements. Unused DD statements should be provided and DUMMY'd, with the exception of the LABEL DD which should only be provided if Bypass Label Processing (BLP) is being used. The LABEL DD should not be DUMMY'd.

    0C4 Errors

    DO NOT specify RENT or REUS when binding the program. The MSGLG610 program is NOT re-entrant and you will get an 0C4 ABEND very early in the MSGLG610 program code if you specify either of these options.

    0C4 errors are much less of a problem with the MSGLG610 version of the program because of its ability to assign artificial message IDs to messages without recognizable message IDs and its ability to aggregate similar messages. This results in almost no unknown messages.

    0C4 errors (protection exception) usually result when the program finds text strings that adhere to the rules for message IDs but are not really message IDs. The number of real, unique message IDs is relative small and the tables and index structures in the program are sized accordingly to reduce search times. When the program encounters text strings that adhere to the rules for message IDs but are not message IDs, the number of unique combinations is usually greater than the program can handle in its index structures. When the program runs out of index blocks, an 0C4 results. The normal situation where the program encounters more valid message IDs than it can place in its tables does not result in a similar exhaustion of the index blocks because the amount of uniqueness is so much less.

    0C9 Errors

    0C9 errors (fixed point divide exception) are much less common, but they too can be caused by invalid input. Invalid time stamp fields have been known to cause this error because message rate computations are based on them. Translation of time stamps to internal format are protected from invalid characters, but result in zero time stamps. In isolated instances, this is satisfactory because a previously encountered time stamp is substituted. However, for this error handling strategy to work, some number of valid timestamps must be received. If all of the time stamps received are zero, 0C9's can result.

    706 Errors

    706 errors typically occur when you initially install the program and received the program load library over the IBM network rather than from a distribution tape.

    A 706 abend is the result of the MVS loader marking the load module as non-executable. The most likely cause is damage to the load module from improper handling of an MVS partitioned data set on a VM system. The members of an MVS load library must be unloaded using IEBCOPY prior to their transmission and possible storage on a VM system. When the members are returned to an MVS system, they must be reloaded into a partitioned data set using IEBCOPY prior to any attempted execution.

    4-digit year fields

    If HCFORMAT=CENTURY was specified on the CONSOLxx HARDCOPY statement, the 'YY' field will be 4-digits ('YYYY') in length. This will cause all fields to the right of the datestamp field to be mis-aligned and cause processing errors. Please see Appendix E. "Processing of SYSLOGs with 4-digit year fields" for a discussion of how to compensate for this condition.

    Zero averages

    The code designed to prevent division by zero exceptions (0C9 errors) will set the result of the unperformed division to zero. This will manifest itself as zero averages.

    Note that the attempt to divide by zero (and the subsequent zero average) is caused by either null input (a log containing no records) or a log in which the program is unable to recognize records, messages or commands. Fixing the input condition will fix the symptoms.

    Loss of Time-Related Reports

    If the "Message Distribution For Busy Intervals report", "Message Rate graphs" and "Instantaneous Message Rate graph" do not appear or are only a couple of lines in length, this indicates that the program is not finding what it recognizes as valid timestamps. This is usually no more than a field alignment problem.

    Messages Not Being Reported As Suppressed

    The Message Analysis Program reports as suppressed messages which have turned on one or more of the following MPF Request Flags:

    • Message not queued to any console

    • Message suppressed by subsystem

    • Message suppressed by exit (MPF exit)

    • Message suppressed by MPF (list entry)

    Messages which have been removed by setting the following flags:

    • Message was deleted

    • Message forced to hardcopy only

    • Message was issued hardcopy only
    are not considered suppressed by the Message Analysis Program.

    Please note that it is also possible for an MPF exit to override the message suppression indicated by an MPF list entry.

    Message Processing "Rules of Thumb"

    General Rules

    1. Delay has its price

      • It is always cheaper to get rid of something early in a path than late in a path

    2. Sophistication has its price

      • Always use the least sophisticated facility that will do the job
        • More primitive facilities (less functionally rich and usually more difficult to program to) will almost always have much less overhead than sophisticated, easy to use facilities.
      • Removal of a message from the programming that issues it is always cheaper than anything else
      • Deletion of a message is always cheaper than suppressing it
      • Suppression of a message is always cheaper than displaying it
      • Suppression of a message is always cheaper than automating it
      • Use easy to program facilities when automation is volatile; use efficient facilities once automation is stable

    Specific Rules

    1. MPF occurs early in the path of a message while NetView occurs late in the path of a message

    2. It is always cheaper (though sometimes more difficult) to perform functions in MPF

      • It is cheaper to suppress messages (by message ID) in MPF because the search mechanism is far more efficient than NetView's

      • It is cheaper to suppress messages (by fields other than the message ID) in an MPF Exit than doing it in the NetView Message Automation Table

      • It is cheaper to automate messages in an MPF Exit than doing it in any NetView facility

    3. It is always easier (though more costly) to perform functions in NetView.

      • If messages are being selectively suppressed (based on fields other than the message ID), it is easier to specify this in the NetView Message Automation Table than programming it in an MPF Exit

    4. The fewer the messages you push through the NetView Message Automation Table the better.

    5. It is cheaper to perform a function entirely within the NetView Message Automation Table than using the table to invoke a CLIST, a REXX EXEC or a command processor

    6. If you can't perform the function entirely with the Message Automation Table, its cheaper to do it in a NetView command processor than doing it in a CLIST or a REXX EXEC

    7. REXX is easier to use and slightly more efficient than NetView CLIST

    Notes

    • MPF

      1. MPF executes in the address space of the message issuer

        • MPF execution is charged to the issuer of the message

        • MPF Exit has access to all of the message issuer's control blocks

      2. MPF deals with the original message, not a copy

        • MPF processing is performed synchronously, in-line as part of the message's path

        • MPF can affect the appearance of a message regardless of what presentation service
          • Multiple Console Support (MCS) local consoles
          • Extended Multiple Console Support (EMCS) consoles, including:
            • JES3 (RJE/NJE consoles)
            • NetView
            • TSO Operator
            • Spool Data Search Facility (SDSF)
            • hardware System Console / Hardware Management Console (HMC)
          • SNA Multiple Console Support (SMCS) consoles, including consoles attached through TCP/IP
          • subsystem consoles
          is ultimately used to display the message
      3. VTAM messages sent (to NetView) through the VTAM Programmable Operator Interface (POI) are not passed through or seen by MPF.
      4. CICS and IMS Master Terminal Operator (MTO) messages sent (to NetView) through NetView's Terminal Access Facility (TAF) are not passed through or seen by MPF.
      5. Don't perform I/O out of MPF Exits
      6. Don't issue the WAIT SVC or any SVC that uses WAIT in an MPF Exit
    • NetView

      1. NetView executes in its own address space

        • NetView execution is charged to the NetView address space

        • NetView has no access to the message issuer's control blocks

      2. NetView deals with a copy of a message, not the original

        • NetView processing is performed asynchronously, out-of-line with the message's normal path

        • NetView can only affect the appearance of messages presented on a NetView console; it cannot effect the appearance of messages being presented on MCS or JES3 consoles or TSO Operator terminals

    • Getting rid of IEF196I wrappered IGD allocation messages

      Ever notice when you start something SUB=MSTR (like an SA NetView) you get lots and lots of IEF196I wrapper WTOs showing many, many other WTOs? Mostly IGD (SMS) related allocation messages? Makes your syslog really hard to wade thru. Well the trick of stopping these WTOs from getting issued is to use MSGLEVEL=(1,0) on the JOB card (the 0 does the trick). But how do you put a JOB card on a started task proc that runs SUB=MSTR? Simple. Well, not so simple since it took me a few days to remember this. Just start your SA NetView with: S netview,SUB=MSTR,MSGLEVEL=(1,0) That's it, no proc change required. You still get some IEF196I messages but almost all of the IGD related ones now never get issued.

    Message Management "Rules of Thumb"

    1. Systems should be managed, not run.

    2. The system should be managed by exception.

      • Don't display a message unless you are requesting the operator to do something or you're answering a specific query.

      • Allow the operator to request only the information he or she is interested in.

      • Don't answer a query with more information than the operator asked for.

      • Responses to operator queries should only go back to the operator that requested the information.

      • Be sure to flag operator responses as operator responses.

      • Don't ask the operator to do something unless it is important.

      • Don't ask the operator to do something unless it truely requires a human to do it (mount a tape, load paper, etc.).

      • Don't ask the operator to do something that he or she does not have the expertise to do. Only put out choices which the operator can be reasonably expected to understand.

    3. If you have to interact with the operator, do it in as few interactions as possible. Don't be overly verbose. At the same time, do not use obscure jargon.

    4. If an operator decision is required, list the options in increasing order of severity.

    5. If a message must be issued, don't assume that it will be read by a human -- it may be read by various forms of automation.

    6. Direct the message to the appropriate decision maker, not to the world. Route messages functionally. Keep unnecessary messages off of the Master Console.

    7. Don't violate the Principle of Least Astonishment.

      "It didn't work the way I expected it to."

    8. Consider what the operator has to do to handle your message:
      • Retention/erasure of action messages
      • Retrieval of action messages
    9. Be sure to delete (DOM) action messages when the action has been taken or the situation requiring the message no longer exists.
    10. Multiple single line messages should be issued as a single multi-line message.
    11. Don't use WTORs as a way to do command entry.
    12. Don't use Write To Log (WTL); use WTO with "Hardcopy Only" specified.
    13. Be sure to document which messages are:

      • new

      • changed (and how)

      • deleted

From the example above, just specify MAP.OBJ(MSGLG610) on the "Other Partitioned Data Set: Data Set Name" line. Specify LET,LIST,MAP on the "Linkage editor/binder options" line and press ENTER. DO NOT specify RENT or REUS -- the MSGLG610 program is NOT re-entrant and you will get an 0C4 ABEND very early in the MSGLG610 program code. You will be shown the results of the binding step and the output of the process will be placed into a LOAD data set which may have the name MAP.LOAD or userid.LOAD depending on your installation's options. You should then alter the STDJES2 or STDJES3 job control language data sets to point to this LOAD library and then you should be ready to run the program.

    The ABENDTBL.TXT, COMPID.TXT, MSGIDS.TXT and MSGTYPE2.TXT files are text files and must be uploaded to your MVS system in -ASCII- form. All four of the files should be placed into an MVS fixed-block (FB) partitioned dataset with an LRECL of 80. It is critical that the member names used be ABENDTBL, COMPID, MSGIDS and MSGTYPE2.

    Appendix D. LOGWTR procedure for creating a log tape

    To create a log tape for analysis: 

    from an MVS console, issue: WRITELOG class
    where "class" is a one-character JES output class identifier. Do -not- use a JES output class identifier that is used for printing -- or you will wind up printing the log rather than transfering it to tape! 
    from an MVS console, issue: S XWTR,3480,volser,class
    where "3480" indicates a 3480 tape device (use 3420 if you don't have 3480's). Instead of "3480", you can specify an actual tape unit address, such as "282". "volser" is the volume serial number (name of the tape) that you want the log written to. "class" is the one-character JES output class identifier you specified on the WRITELOG command.

    Note that XWTR is the name of the IBM-supplied external writer procedure. If your installation has changed the name, you will have to find out what the new name is and use that.

    I recommend cloning the IBM-supplied external writer procedure (giving it a new name like LOGWTR) and changing the procedure to use a standard labelled tape. That way all of the volser and data set characteristics information is stored on the tape with the data. I suggest using a standard data set name of SYSLOG.

//* REFER:  PLACE THIS PROCEDURE IN SYS1.PROCLIB
//*
//* DOC:    THIS PROCEDURE IS A MODIFIED VERSION OF THE STANDARD
//*         IBM-SUPPLIED EXTERNAL WRITER PROCEDURE "XWTR".
//*
//*         THE DCB PARAMETERS ON THE IEFRDER DD STATEMENT
//*         HAVE BEEN CHANGED TO SPECIFY A STANDARD LABELLED TAPE
//*         AND A DATA SET NAME OF SYSLOG.  THE BLKSIZE AND
//*         BUFL PARAMETERS HAVE BEEN INCREASED FROM 13030 TO
//*         32000 TO MAKE BETTER USE OF THE SPACE ON THE TAPE

//*        .    *   BUFL    = 32000
//*        .    *   BUFNO   = 2
//*        .    *   RECFM   = VBM
//*
//IEFPROC  EXEC  PGM=IASXWR00,PARM='PA',REGION=128K,

    Appendix E. Processing of SYSLOGs with 4-digit year fields

    You can request that SYSLOGs be produced with a 4-digit year field instead of the standard 2-digit year field. Use of the 4-digit year field shifts all of the fields to the right of the year field rightward by two character positions. Use of the 4-digit year field is selected by specifying 

    HCFORMAT=CENTURY
    on the CONSOLxx member HARDCOPY statement.

    If you wish to process a SYSLOG with 4-digit year fields with this program, the following field alignment statements may be used to put the SYSLOG into a form that the program can process: 

    LOGDATE(21) or LOGYEAR(21) and LOGDAY(23)
LOGTIME(27)
LOGJOB(39)
LOGREQFL(48)
LOGMSG(57)

    The YEAR4 option provides a convenient shorthand for specifying these individual input options.

    Appendix H. Glossary

    Action Messages
    An unsolicited message asking the operator to do something, typically to take a physical action (such as mounting a tape or adding paper to a printer).
    Busy Interval
    A busy interval is a period of time during which at least one message occurred.
    Decision Messages
    An unsolicited message asking the operator to make a decision , usually from a list of options.
    Known Messages
    Messages which have been classified in the MSGTYPE2 table.
    MLG Messages
    In a JES3 environment, the destination class for the log. Messages sent to MLG are explicitly written to the log only.
    Re-routed Messages
    In a JES3 environment, the device-related messages which JES3 explicitly re-directs to the functional consoles responsible for the device.
    Solicited Messages
    Messages produced in response to an operator command; a command response. These messages are typically returned to the console from which the command was entered.
    Suppressible Messages
    Messages which have been classified as suppressible in the MSGTYPE2 table. Suppressible messages have been judged to have no important information content.
    Unknown Messages
    Messages which have not been classified in the MSGTYPE2 table.
    Unseen Messages
    Messages which are not visible to the operator.
    Unsolicited Messages
    Messages produced asynchronously; usually indicating some sort of state change. Operator action may or may not be indicated. These messages are typically routed to consoles by attributes such as route codes (which indicate functional area) and descriptor codes (which describe the importance of the message).
    Unsuppressible Messages
    Messages which have been classified as unsuppressible in the MSGTYPE2 table. Unsuppressible messages have been judged to have important information content which should not be lightly ignored.
    Write-To-Log (WTL) Messages
    Messages which have been written to the log using the WTL macro. Such messages receive no formatting. Use of WTL should be avoided.
    Write-To-Programmer Messages
    Messages which have been explicitly routed to Route Code 11, which is reserved for sending messages to a job's job log. Such messages are typically of no interest to anyone other than the programmer they have have been issued to.
